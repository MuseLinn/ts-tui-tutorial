## 一、执行摘要

本方案基于对 `ts-tui-tutorial` 代码库的全面健康检查，以及对 Ink/React TUI 生态中优秀项目（Rustlings、Cloudflare Wrangler、Prisma CLI、cfpush、lrn）的深入研究编制而成。

**核心发现**：项目目前已完成 MVP 阶段的功能验证（12 节课、实时 tsc 校验、多行编辑器、21 个单元测试通过），但在**架构稳定性、运行时安全、测试覆盖、可维护性**方面存在 5 大类 30+ 项问题，其中 **4 项 Critical 风险** 可能在真实用户场景中导致崩溃或数据丢失。

**维护目标**：
1. 消除 Critical 级运行时风险
2. 建立可长期演进的内容与架构体系
3. 补齐 UI/集成测试覆盖
4. 构建生产级错误处理、遥测与配置机制

## 二、现状诊断

### 2.1 架构与设计问题

| 问题 | 位置 | 严重性 | 影响 |
|------|------|--------|------|
| **Reducer 不纯** | `AppContext.tsx:132-165` | 🔴 Critical | `saveProgress()` 被直接调用在 reducer 内部，违反纯函数原则，可能导致不可预测的状态回滚与重复保存 |
| **Render 中 dispatch** | `Layout.tsx:48-62` | 🔴 Critical | 在 React 渲染路径中直接调用 `dispatch({COMPLETE_LESSON})`，可引发无限循环或并发状态更新错误 |
| **全局单例耦合** | `ExerciseEngine.ts:4` | 🔴 Critical | 模块级 `const tscService = new TscService()` 使引擎与具体实现硬耦合，单元测试难以 mock |
| **God Object State** | `types.ts:111-124` | 🟡 High | `AppState` 混合导航、UI 临时状态、用户输入、持久进度，导致任何微小改动都可能影响全局 |
| **布局组件过度耦合** | `Layout.tsx` | 🟡 High | 同时承担布局渲染、全局键盘路由、自动完成逻辑三重职责 |

### 2.2 用户体验问题

| 问题 | 位置 | 严重性 | 影响 |
|------|------|--------|------|
| **无顶层错误边界** | `cli.tsx` | 🔴 Critical | Ink TUI 崩溃后终端可能卡在 raw mode，用户无法恢复 |
| **假帮助按钮** | `Header.tsx:20` | 🟡 High | 界面显示 `[?] 帮助`，但代码中无任何 `?` 键处理器 |
| **导航行为不一致** | `Layout.tsx:26-45` | 🟡 High | `→` 箭头在 `lastResult=correct` 时行为突变，从"切换视图"变成"下一步"，用户无法建立稳定心智模型 |
| **缺少加载态** | `AppContext.tsx` | 🟡 Medium | 进度异步加载时，界面先以默认状态闪烁渲染 |
| **理论内容无滚动** | `TheoryView.tsx` | 🟡 Medium | 长文本直接溢出终端可视区 |

### 2.3 代码质量问题

| 问题 | 位置 | 严重性 | 影响 |
|------|------|--------|------|
| **未处理 RegExp 异常** | `ExerciseEngine.ts:47` | 🔴 Critical | `contains` 模式的 `new RegExp(pattern)` 可抛 `SyntaxError`，直接崩溃 |
| **进度数据无运行时校验** | `ProgressRepo.ts:21,40` | 🔴 Critical | `JSON.parse` 后直接 `as UserProgress`，磁盘损坏/用户手动编辑会导致非法状态传播 |
| **内存泄漏风险** | `TscService.ts:43` | 🟡 High | 每次校验新建 `ts.createProgram` 且不释放，长时间学习会话中内存持续增长 |
| **自动保存 race condition** | `AppContext.tsx:207-218` | 🟡 High | `useEffect` 依赖整个 `[state]` 对象，用户每敲一个键就重置 30s 定时器，可能永远触达不到保存 |
| **不安全类型断言** | `AppContext.tsx` | 🟡 Medium | `(step?.type as AppState['currentView'])` 出现 4 次，新增 step 类型时可能引入静默 bug |

### 2.4 可维护性问题

| 问题 | 位置 | 严重性 | 影响 |
|------|------|--------|------|
| **课程数据零验证** | `curriculum/index.ts` | 🟡 High | 静态数组无运行时校验，prerequisite ID 可能指向不存在的课程；已安装 `zod` 但未使用 |
| **硬编码中文文本散落** | `ExerciseEngine.ts` + `ExerciseView.tsx` | 🟡 Medium | 成功/失败提示重复且硬编码，无法支持 i18n 或统一文案管理 |
| **重复状态重置逻辑** | `AppContext.tsx` | 🟡 Medium | `SET_LESSON`/`NEXT_STEP`/`PREV_STEP`/`SET_VIEW` 中重复 4 遍相同的 UI reset 代码 |
| **魔术数字遍地** | 多文件 | 🟡 Low | Sidebar width=18、Editor height=10、Auto-save=30000ms 等 |
| **无用依赖** | `package.json` | 🟡 Low | `fuse.js`、`meow`、`ava` 已安装但源码未使用 |

### 2.5 测试覆盖缺口

| 缺口 | 严重性 | 说明 |
|------|--------|------|
| **UI 组件零测试** | 🔴 Critical | `CodeEditor`、`ExerciseView`、`QuizView`、`Sidebar`、`Layout` 等全部未测 |
| **无集成测试** | 🔴 Critical | 编辑器按键 → 状态更新 → `validateExercise` → `TscService.check()` 的完整链路未测 |
| `ProgressRepo` 未测 | 🟡 High | 文件 I/O 逻辑无任何覆盖 |
| `AppContext` reducer 未测 | 🟡 High | 导航、完成、保存等核心 action 未测 |

## 三、设计目标与原则

### 3.1 目标

1. **稳定性优先**：在 1-2 天内消除所有 Critical 级运行时崩溃风险。
2. **可测试性**：解耦全局单例，让引擎、服务、组件均可独立测试。
3. **用户体验**：修复导航不一致性，补齐帮助系统、加载态、错误边界。
4. **可维护性**：建立配置层、内容校验层、字符串字典层，消灭魔术值与硬编码。
5. **测试覆盖**：补齐 UI 组件测试和集成测试，达到关键路径全覆盖。

### 3.2 原则

- **Reducer 必须是纯函数**：所有副作用（文件 I/O、定时器、网络）只能存在于 `useEffect` 或 middleware 中。
- **Schema 优先**：任何从磁盘加载的数据（进度、课程、配置）必须经过运行时 schema 校验（使用已安装的 `zod`）。
- **Fail-safe**：状态文件损坏时， graceful degradation 到"新用户"体验，而非崩溃。
- **Never block for telemetry**：遥测、崩溃上报必须是 fire-and-forget，不得阻塞用户交互。
- **Content != Code**：课程内容必须与渲染代码完全解耦，使用声明式 YAML/TOML 驱动。

## 四、分阶段维护路线图

### Phase 1：紧急修复（1-2 天）
消除所有 Critical 级运行时风险。

| 任务 | 改动文件 | 具体方案 |
|------|----------|----------|
| 修复 reducer 不纯问题 | `AppContext.tsx` | 将 `saveProgress()` 从 reducer 中彻底移除，改为 `useEffect` 监听 `[progress]` 字段变化时触发保存 |
| 修复 render 中 dispatch | `Layout.tsx` | 将 `COMPLETE_LESSON` 逻辑移入 `useEffect`，依赖 `[currentLessonId, currentStepIndex, lastResult]` |
| 修复未处理 RegExp 异常 | `ExerciseEngine.ts` | 用 `try/catch` 包裹 `new RegExp(pattern)`，异常时返回 `passed: false` + 友好提示 |
| 添加顶层错误边界 | `cli.tsx` | 添加 `try/catch` 包裹 `render(<App />)`，`catch` 中调用 `process.stdin.setRawMode(false)` 恢复终端 |
| 解耦 TscService 单例 | `ExerciseEngine.ts` | `validateExercise(step, code, tscService = defaultTscService)`，默认参数注入 |

### Phase 2：数据与持久化加固（2-3 天）
建立数据完整性与安全的护城河。

| 任务 | 改动文件 | 具体方案 |
|------|----------|----------|
| 进度数据 Schema 校验 | `ProgressRepo.ts` | 使用 `zod` 定义 `UserProgressSchema`，`loadProgress` 时 `safeParse`，失败则回退默认值 |
| 课程数据校验 | `curriculum/index.ts` | 启动时断言：所有 `prerequisites` 必须存在于 curriculum 的 `id` 集合中 |
| 原子写与迁移 | `ProgressRepo.ts` | 实现 `writeFileSync(tmp) + renameSync(tmp, target)`；`UserProgress.version` 字段驱动迁移逻辑 |
| 修复 auto-save race | `AppContext.tsx` | `useEffect` 依赖改为具体字段 `[currentLessonId, currentStepIndex, progress.completedLessons]` |
| 清理无用依赖 | `package.json` | 移除 `fuse.js`、`meow`、`ava` |

### Phase 3：架构重构（3-5 天）
拆分 God Object，建立配置与字典层。

| 任务 | 改动文件 | 具体方案 |
|------|----------|----------|
| 拆分 God Object State | `context/` | 拆分为 `useNavigation()`、`useProgress()`、`useExerciseState()` 三个独立 hook，顶层 `App` 做 orchestration |
| 提取通用 helper | `AppContext.tsx` | 将重复 4 次的 reset 逻辑提取为 `createStepTransitionState()` |
| 建立配置层 | `source/config.ts` | 集中管理 `AUTO_SAVE_INTERVAL_MS = 30000`、`SIDEBAR_WIDTH = 18`、`DEFAULT_COMPILER_OPTIONS` 等 |
| 建立字符串字典 | `source/i18n/zh-CN.ts` | 集中管理所有用户-facing 字符串，支持未来多语言扩展 |
| 引入 logger | `source/utils/logger.ts` | 极简 logger：`debug/info/warn/error`，默认 `warn` 以上输出到 stderr |

### Phase 4：UI/UX 增强（3-5 天）
补齐缺失的交互体验。

| 任务 | 改动文件 | 具体方案 |
|------|----------|----------|
| 实现帮助覆盖层 | `components/HelpOverlay.tsx` | 按 `?` 弹出快捷键说明覆盖层；在 `Layout.tsx` 中绑定 `?` 键 |
| 增加加载态 | `AppContext.tsx` + `Layout.tsx` | `INIT` 完成前渲染 `Loading...` 或启动画面 |
| TheoryView 滚动 | `TheoryView.tsx` | 增加 `scrollOffset` 状态，支持 `j/k` 或 `↑/↓` 滚动长文本 |
| 统一导航行为 | `Layout.tsx` | `→` 始终切换 view；`Tab` 或 `n` 进入下一步；消除条件行为 |
| 内存优化 TscService | `TscService.ts` | 缓存 `ts.Program`，仅在 `code` 变化时重建 |

## 五、架构改进设计

### 5.1 改进后的依赖关系

```
┌─────────────────────────────────────────────────────────────┐
│                        Presentation                          │
│  Layout ──► Sidebar / Header / Footer / HelpOverlay          │
│     │                                                        │
│     ├───► Router ──► TheoryView / CodeView / QuizView        │
│     │                / ExerciseView ──► CodeEditor           │
│     │                                                        │
├─────────────────────────────────────────────────────────────┤
│                      State Layer (Split)                     │
│  useNavigation()  useProgress()  useExerciseState()          │
│       │                 │                │                   │
│       └─────────────────┴────────────────┘                   │
│                         │                                    │
│                  AppContext (orchestrator)                   │
├─────────────────────────────────────────────────────────────┤
│                        Engine / Services                     │
│  LessonEngine  ExerciseEngine  TscService                    │
│       │              │              │                        │
│       └──────────────┴──────────────┘                        │
│                         │                                    │
│                   ProgressRepo  Config  I18n  Logger         │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 关键设计决策

#### 1) Reducer 纯函数化
所有副作用（文件保存、定时器、日志输出）必须从 `appReducer` 中移除，迁移到 `useEffect`。reducer 只负责计算新状态。

#### 2) TscService 依赖注入
```ts
// 默认注入，测试可替换
export function validateExercise(
    step: ExerciseStep,
    userCode: string,
    tscService: TscService = defaultTscService,
): ExerciseResult { ... }
```

#### 3) Schema 校验防线
利用已安装的 `zod`，为所有外部输入建立 schema：
- `UserProgressSchema` —— 加载进度时 `safeParse`
- `LessonSchema` —— 启动时验证 curriculum 数组
- `AppSettingsSchema` —— 加载配置时验证

#### 4) Ink 错误边界策略
Ink 内部已有 `ErrorBoundary`，但我们在 `cli.tsx` 外再包一层 `try/catch`：
```ts
try {
    render(<App />);
} catch (err) {
    if (process.stdin.isTTY) {
        process.stdin.setRawMode(false);
    }
    console.error('Fatal error:', err);
    process.exit(1);
}
```

#### 5) 配置集中化
新增 `source/config.ts`：
```ts
export const CONFIG = {
    AUTO_SAVE_INTERVAL_MS: 30_000,
    SIDEBAR_WIDTH: 18,
    CODE_EDITOR_HEIGHT: 12,
    PROGRESS_DIR: path.join(os.homedir(), '.ts-tui-tutorial'),
    DEFAULT_COMPILER_OPTIONS: { target: ts.ScriptTarget.ES2022, strict: true, noEmit: true },
} as const;
```

#### 6) 课程热重载准备
当前课程写在 `.ts` 文件中，不利于非开发者更新。中期应将课程迁移到 YAML：
```yaml
format_version: 1
lessons:
  - id: 01-variables
    title: 变量与类型注解
    steps:
      - type: theory
        content: |
          ## 什么是变量？
          ...
```
开发模式下通过 `fs.watch` 监听 YAML 变化，自动重新解析并更新状态。

## 六、长期维护机制

| 机制 | 说明 |
|------|------|
| **版本化进度数据** | `UserProgress.version = 1` 已存在。`ProgressRepo` 增加 `migrate()` 函数，自动将旧版本数据升级到最新格式 |
| **课程热重载** | 开发模式加载 `curriculum.yaml`，生产模式内嵌编译后的 JSON。内容更新无需重新编译 TypeScript |
| **A/B 课程插槽** | 在 `curriculum/index.ts`（或 YAML）中保留 `experimentalLessons` 数组，方便灰度发布新课程 |
| **调试模式** | 增加 `--verbose` / `--debug` CLI 参数，输出 `TscService` 诊断详情和 reducer action trace |
| **遥测与崩溃上报** | 参考 Wrangler 模式：Sentry 崩溃报告默认关闭，首次崩溃时弹窗请求用户同意；遥测事件使用 fire-and-forget fetch；PII 脱敏 |
| **CI 检查清单** | 在 GitHub Actions 中增加：1) `tsc` 零错误 2) `vitest` 全绿 3) `curriculum` prerequisite 完整性校验 4) `zod` schema 校验通过 |

## 八、下一步行动建议

如果你希望立即进入实施阶段，建议按以下顺序执行：

1. **Phase 1 紧急修复**：这是最高优先级，直接关乎用户是否会遇到崩溃。
2. **Phase 2 数据加固**：在吸引真实用户之前，必须保证进度数据的安全。
3. **Phase 5 补齐 CodeEditor + ExerciseView 测试**：编辑器是用户最频繁交互的组件，必须被测试覆盖。

其余 Phase 3（架构重构）和 Phase 4（UX 增强）可以在前三个阶段稳定后逐步推进。

如需进入实施，请告知从哪个 Phase 开始。
